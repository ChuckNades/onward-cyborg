Get your Apple calendar link 🔗

Cyborg reads your calendar through a public ICS link — a read-only web address for a calendar. It can only read; it can never change your calendar. No passwords or API keys are stored anywhere. Here's how to get the link from iCloud (do this on your phone or Mac):

On your iPhone, open Calendar → tap the calendar's ⓘ (Edit) → turn on Public Calendar.
Tap Share Link → Copy Link. It looks like webcal://p XX-caldav.icloud.com/published/2/…
Change the start from webcal:// to https://. That's your ICS URL. Save it somewhere you can paste it later.
Want more than one calendar (e.g. "Family" and "School")? Repeat for each — you'll give each one its own colour later.
pasting the https://… link into a browser downloads a .ics file full of BEGIN:VEVENT lines. That's the calendar data Cyborg will read.
🛟 If it didn't… there's no "Public Calendar" switch You're probably on a calendar you don't own (a subscribed/holiday calendar). Public sharing only works on calendars you created. Make a new calendar, move/add events to it, and share that one. On a Mac you can also do this in the Calendar app: right-click the calendar → Share Calendar → tick Public Calendar.
🟡 Keep the link private-ish. Anyone with the link can see the calendar. Don't post it publicly. You can rotate it any time by toggling Public Calendar off/on.

The Golden Safety Rules 🛡️

GOLDEN RULE #1 — POWER OFF FIRST. Always unplug the Pi before you touch, add, remove, or check ANY pin/wire on the 40-pin header. Plug power back in only after a grown-up has checked every wire against the diagram. A loose wire brushed across a powered header is the easiest way to kill the Pi.
GOLDEN RULE #2 — NEVER bridge 5 V to 3.3 V or GND. Count pins twice against the diagram before every connection. Wire one mission at a time, check it, then move on.

Meet the 40-pin header (only the pins we use)

Finding pin 1: it's the only pin with a square solder pad (all others are round), at one end of the header. Pins go in pairs down the header: 1&2, 3&4, 5&6… Odd numbers (1,3,5…) are one row; even (2,4,6…) the other. Always anchor on the square pad = pin 1.

   ODD ROW (1,3,5…)            EVEN ROW (2,4,6…)
        3V3  [ 1]■  [ 2]   5V
 RTC▶ GPIO2  [ 3]   [ 4]   5V   ◀ FAN +
 RTC▶ GPIO3  [ 5]   [ 6]   GND  ◀ FAN −
       GPIO4 [ 7]   [ 8]   GPIO14
  RTC▶  GND  [ 9]   [10]   GPIO15
BTN▶ GPIO17  [11]   [12]   GPIO18
      GPIO27 [13]   [14]   GND  ◀ BUTTON −
        ...      (pins 15–40 mostly unused) ...
■ = square pad = pin 1.  FAN=cooling fan · RTC=DS3231 clock · BTN=power button.
Speaker is NOT a pin device — it plugs into the 3.5 mm audio jack.
top-down shot of your real Pi header with pins 1, 4, 6, 11, 14 flagged with tape.

🚀 Weekend 1 — "Boot Cyborg Up" (v1)

By the end of this weekend you'll have a working portrait wall calendar, themed as Cyborg, that survives Wi-Fi drops and shouts BOOYAH!. That's the whole, complete, celebrate-it build. (Weekend 2 is an optional level-up.)

Mission 1 — Flash the card & first boot

Role lead: 💻 Commander of Code · Difficulty ★★☆ · ~45 min

We put the operating system (the Pi's "Windows") onto the microSD card, then boot Cyborg for the first time and get it on Wi-Fi.

On your other computer, download & install the Raspberry Pi Imager from raspberrypi.com/software. Put the microSD into the card reader.
Open Imager → Choose Device: Raspberry Pi 3 → Choose OS: Raspberry Pi OS (other) → Raspberry Pi OS Lite (32-bit). (Lite = no desktop, perfect for our light little Pi.)
Choose Storage: your microSD. Then click the ⚙️ gear (Edit Settings): set hostname cyborg; enable SSH (password); set username pi + a password you'll remember; enter your Wi-Fi name + password + your country; set your locale/timezone.
Click Write. Wait for "Write Successful". Eject the card and put it into the Pi's slot (contacts facing the board).
Plug in keyboard + the monitor (HDMI) for now. Plug in Pi power last. The green light blinks — Cyborg is booting!
Log in as pi with your password. Then update everything: 
sudo apt update && sudo apt full-upgrade -y
sudo apt install -y git python3-venv python3-pip
ping -c 3 raspberrypi.com
you see a login prompt that says cyborg login:, you log in, and the ping shows replies (Cyborg is on the internet). BOOYAH — Cyborg is alive!
🛟 If it didn't… no Wi-Fi / ping fails Re-check the Wi-Fi name + password (case-sensitive) you typed in the Imager gear. Run sudo raspi-config → System Options → Wireless LAN to re-enter them, then sudo reboot. Make sure you're on a 2.4 GHz network — the Pi 3B's Wi-Fi doesn't see some 5 GHz-only networks.
🛟 If it didn't… black screen / no boot Re-seat the microSD. Re-flash the card (a bad write is common). Use the official Pi power supply — under-powered chargers cause boot loops. If you see a rainbow square, the card didn't flash correctly.

🎖️ Badge earned: ⚡ Spark of Life

Mission 2 — Keep Cyborg cool (heatsinks + fan)

Role lead: 🔧 Master of the Mount & 🔌 Captain of Cables · ★★☆ · ~20 min · CORE
POWER OFF FIRST. Unplug the Pi before touching any pin.
Peel and stick the heatsinks onto the two main chips (the big square one is the processor).
Wire the fan: RED (+) → Pin 4 (5 V), BLACK (−) → Pin 6 (GND). Count pins twice. The fan should blow onto the board; the finger-guard faces out.
Fan RED  (+) ───► Pin 4  (5V rail)
Fan BLACK(−) ───► Pin 6  (GND)
Always-on power rail — NOT a switch. Fan ≤ 0.1 A, guard fitted.
Grown-up double-checks the two pins against the diagram, then power on. Check the temperature: 
vcgencmd measure_temp
on power-up you feel air moving, and measure_temp reads well under 70°C (e.g. temp=48.3'C). Cool as a cyborg. 😎
🛟 If it didn't… fan doesn't spin Power off. The connector is on the wrong pins or backwards. The fan's + must be on Pin 4 and − on Pin 6 — these are right next to each other near pin 1. Some fans only spin reliably from the 5 V rail (pins 2/4); we use pin 4. Re-seat firmly.

🎖️ Badge earned: ❄️ Cool Customer

Mission 3 — Cyborg's heartbeat (DS3231 real-time clock)

Role lead: 🔌 Captain of Cables · ★★★ · ~30 min · CORE (Level-Up)

The Pi forgets the time when it's unplugged. The DS3231 is a tiny clock with its own battery so Cyborg shows the right time the instant it powers on — even before Wi-Fi connects.

POWER OFF FIRST. And note: RTC VCC goes to Pin 1 (3.3 V) — NEVER Pin 2 or 4 (those are 5 V and will damage it).
Wire the DS3231 (four wires):
RTC VCC ─► Pin 1  (3V3)     ⟵ NOT 5V!
RTC GND ─► Pin 9  (GND)
RTC SDA ─► Pin 3  (GPIO2 / I²C data)
RTC SCL ─► Pin 5  (GPIO3 / I²C clock)
Grown-up checks the 4 wires. Power on, log in, then install the I²C tools and turn I²C on: 
sudo apt install -y i2c-tools
sudo raspi-config nonint do_i2c 0
sudo reboot
After it reboots, log in and look for the clock: 
sudo i2cdetect -y 1
You should see 68 in the grid (its address).
Tell Linux to use it, then save the current (Wi-Fi-synced) time onto the chip: 
sudo apt install -y python3-smbus
echo "dtoverlay=i2c-rtc,ds3231" | sudo tee -a /boot/firmware/config.txt
sudo reboot
# after reboot:
sudo hwclock -r        # read the RTC
sudo hwclock -w        # write current time to the RTC
sudo i2cdetect -y 1 shows 68, and sudo hwclock -r prints the correct date and time. Cyborg's heart is ticking. ❤️
🛟 If it didn't… nothing at 0x68 Power off and recheck: SDA on Pin 3, SCL on Pin 5, VCC on Pin 1 (not 5 V!), GND on Pin 9. A swapped SDA/SCL or VCC on a 5 V pin are the usual culprits. Make sure you ran do_i2c 0 and rebooted.

ℹ️ Because the clock uses GPIO3, the "short GPIO3 to wake a halted Pi" trick isn't available — that's fine, our power button (Mission 9) uses GPIO17 instead.

🎖️ Badge earned: ❤️ Heartbeat

Mission 4 — Install Cyborg's brain (the code)

Role lead: 💻 Commander of Code · ★★☆ · ~35 min · CORE

Now we put Cyborg's software on the Pi and give it your calendar link(s). First we'll run it by hand to prove it works, then later missions make it start automatically.

Get the project folder onto the Pi. Pick the line that matches how you have it, then run the ls check: 
# Option A — it's on a USB stick (the part after /media/pi/ is the stick's name):
cp -r /media/pi/*/onward-cyborg ~/onward-cyborg

# Option B — copy from another computer on the same Wi-Fi (run this ON that computer):
#   scp -r ~/forge/onward-cyborg  pi@cyborg.local:~/onward-cyborg

# Option C — you host it in git:
#   git clone <your repo url> ~/onward-cyborg

cd ~/onward-cyborg
ls cyborg_core      # you should see config.py  server.py  service.py …
Make Cyborg's settings file from the example, then open it to edit: 
cp config.example.toml config.local.toml
nano config.local.toml
In the editor, set home_tz to your timezone, and for each calendar paste your https://… ICS link into url and pick a color. Example: 
[service]
home_tz = "America/Chicago"
port = 8765

[[calendars]]
id = "family"
name = "Family"
url = "https://p XX-caldav.icloud.com/published/2/YOUR-FAMILY-LINK"
color = "#4C78A8"

[[calendars]]
id = "school"
name = "School"
url = "https://p XX-caldav.icloud.com/published/2/YOUR-SCHOOL-LINK"
color = "#F58518"
Save in nano with Ctrl+O, Enter, then Ctrl+X.
Make a little Python sandbox and install Cyborg's two helper libraries: 
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
Run Cyborg by hand: 
.venv/bin/python -m cyborg_core --config config.local.toml
Leave it running. On the Pi (or another computer on the same Wi-Fi) open a browser to http://cyborg.local:8765/ (or http://<the Pi's IP>:8765/).
http://cyborg.local:8765/health opens and shows a page of status fields (raw JSON) including "fetch": {"ok": true …} — that means Cyborg reached your calendar. The calendar page itself may say "waking up / syncing" for now — that's expected: Cyborg needs the memory stick from Mission 5 before it can save and show events. Press Ctrl+C to stop it for now. BOOYAH — the brain works!
🛟 If it didn't… /health shows "fetch": {"ok": false} Cyborg couldn't read your calendar. Check your url lines really start with https:// (not webcal://), and paste one link straight into a browser to confirm it downloads a .ics file. The fetch.last_error field on /health (and any red line in the terminal) tells you exactly what went wrong. (A "waking up / syncing" calendar screen at this stage is normal — that's fixed in Mission 5.)
🛟 If it didn't… "cyborg.local" won't load Find the Pi's IP with hostname -I and use http://THAT-IP:8765/ instead. Make sure the other computer is on the same Wi-Fi.

🎖️ Badge earned: 🧠 Brain Online

Mission 5 — The memory stick (USB last-known-good cache)

Role lead: 💻 Commander of Code · ★★☆ · ~20 min · CORE

Cyborg saves a copy of the calendar so the screen never goes blank when Wi-Fi drops. We keep that copy on a USB stick instead of the SD card, because writing to the SD card all day, every day would wear it out. (Real engineers protect their storage!)

This ERASES the USB stick. Use an empty one. Double-check you picked the right drive letter below.
Plug the USB stick into the Pi. Find its name: 
lsblk
It's usually sda with a partition sda1 (NOT mmcblk0 — that's the SD card!).
Format it as Linux ext4 and give it the label CYBORG (the setup script looks for this label): 
sudo umount /dev/sda1 2>/dev/null || true
sudo mkfs.ext4 -L CYBORG /dev/sda1
Type y if it asks to proceed.
🛟 If it didn't… "device is busy" or you can't find sda1 Run lsblk again. The USB is the one whose size matches your stick — usually sda1, never mmcblk0 (that's the SD card with the OS!). "Busy" means it's mounted: run the sudo umount line first. If your stick shows as sdb1 instead, use that name everywhere in this mission.
Mount it now to test, make the cache folder, and check the UUID: 
sudo mkdir -p /mnt/cyborg
sudo mount /dev/sda1 /mnt/cyborg
sudo mkdir -p /mnt/cyborg/cyborg-cache
sudo chown -R pi:pi /mnt/cyborg/cyborg-cache
blkid /dev/sda1     # shows UUID=... LABEL="CYBORG"
Run Cyborg by hand again, watch the cache file appear, then check the calendar while it's still running: 
cd ~/onward-cyborg
.venv/bin/python -m cyborg_core --config config.local.toml &
sleep 8
ls -l /mnt/cyborg/cyborg-cache/    # you should see last-known-good.json
Leave it running and open http://cyborg.local:8765/ in a browser — your events should now show. Once you've seen them, stop the test server: 
kill %1
blkid shows LABEL="CYBORG"; after running, a file last-known-good.json appears in /mnt/cyborg/cyborg-cache/; and now the calendar page at http://cyborg.local:8765/ finally shows your events — Cyborg has a place to save them. That's the backup memory. 🧠💾
🛟 If it didn't… no last-known-good.json Cyborg only writes the cache when it has fetched real events, and only writes when something changed. Make sure your calendar links work (Mission 4) and give it ~10 seconds. Check http://cyborg.local:8765/health → the usb section should say writable: true. If it says using_fallback: true, the stick isn't mounted at /mnt/cyborg — re-do the mount step.

🎖️ Badge earned: 💾 Backup Memory

Mission 6 — Stand tall & start on boot (portrait + kiosk)

Role lead: 💻 Commander of Code · ★★★ · ~30 min · CORE

One magic script wires everything to start automatically: it installs the kiosk, rotates the screen to portrait, sets up the night-time screen schedule, mounts your USB by its label, and makes Cyborg boot straight into the calendar. Let's peek first, then run it.

See exactly what it would do (changes nothing): 
cd ~/onward-cyborg
sudo ./scripts/setup.sh --dry-run | less
Press q to quit the preview.
Run it for real (this installs packages and may take a few minutes): 
sudo ./scripts/setup.sh
Give the installed service your calendar settings (the script created a placeholder): 
sudo cp ~/onward-cyborg/config.local.toml /etc/cyborg/config.local.toml
sudo systemctl restart cyborg-fetch.service
systemctl status cyborg-fetch.service --no-pager
Look for active (running).
Reboot into the finished kiosk: 
sudo reboot
after reboot, the Pi automatically shows the full-screen portrait Cyborg calendar — no login, no desktop, no mouse needed. The clock ticks and your events are listed. BOOYAH — that's the appliance!
🛟 If it didn't… screen is sideways Our config rotates "left". If your monitor is mounted the other way, edit /etc/X11/xorg.conf.d/99-cyborg-portrait.conf and change "left" to "right", then sudo reboot.
🛟 If it didn't… boots to a text login, no calendar The graphical kiosk runs as the pi user via auto-login on screen 1. Run sudo systemctl status getty@tty1 (should show the autologin) and cat ~/.xinitrc (should say exec openbox-session). Re-run sudo ./scripts/setup.sh — it's safe to run again. Confirm the fetch service is up: systemctl status cyborg-fetch.
🛟 If it didn't… screen is blank/black after boot Open http://cyborg.local:8765/health from your phone. If that loads, the brain is fine and it's a display issue (check the HDMI cable + that the monitor input is the right HDMI port). If it doesn't load, see Troubleshooting HQ.

🎖️ Badge earned: 🧍 Stand Tall

Mission 7 — Give Cyborg a voice (BOOYAH!)

Role lead: 🔌 Captain of Cables · ★★☆ · ~20 min · CORE

Time for the best part. Cyborg shouts BOOYAH! on three system moments only: when it wakes up, when it successfully refreshes, and when it recovers after being offline. It never reacts to what's in your calendar — that's a promise baked into the code (and tested!). It's also rate-limited so it's a treat, not a nuisance.

Plug the powered speaker into the Pi's 3.5 mm jack and turn it on. Force audio out of that jack: 
sudo raspi-config nonint do_audio 1   # 1 = 3.5mm headphone jack
amixer set Master 90%
On newer Raspberry Pi OS you can instead choose the output in sudo raspi-config → System Options → Audio, then test with speaker-test -t wav -c 2 (Ctrl+C to stop).
Give Cyborg a sound. The repo ships no audio, so we make our own chime right on the Pi (you can swap in a real "Booyah!" recording later). It must live in this local folder — it's never committed to the project: 
sudo apt install -y sox libsox-fmt-mp3
sudo mkdir -p /opt/cyborg-core/web/assets
# make a cheerful two-note chime as booyah.mp3:
sudo sox -n /opt/cyborg-core/web/assets/booyah.mp3 synth 0.18 sine 660 synth 0.28 sine 990 vol 0.6
sudo chown -R pi:pi /opt/cyborg-core/web/assets
Prefer your own voice? Record one and save it to the same path: arecord -d 2 -f cd booyah.wav then sox booyah.wav /opt/cyborg-core/web/assets/booyah.mp3. Want a face for Cyborg too? Drop an image at /opt/cyborg-core/web/assets/cyborg-mark.png — it replaces the placeholder "C".
Make the BOOYAH fire on the next wake-up: 
sudo systemctl restart cyborg-fetch.service
# then reload the kiosk page (or sudo reboot)
when the kiosk loads, you hear your BOOYAH! once. Refresh the page a few times — it stays quiet (the rate-limit), proving it only fires on real system events. 🔊
🛟 If it didn't… no sound Three checks: (1) the file is exactly /opt/cyborg-core/web/assets/booyah.mp3 — open http://cyborg.local:8765/assets/booyah.mp3 in a browser; it should play. (2) The speaker is powered on and the jack is fully seated. (3) Run speaker-test -t wav -c 2 to confirm audio output, and re-run the do_audio 1 step. The kiosk is configured to allow auto-play sound, so once the file and speaker are right, it just works.

🎖️ Badge earned: 🔊 The Voice

Mission 8 — Mount Cyborg on the wall

Role lead: 🔧 Master of the Mount + 🎖️ Mission Control · ★★★ · ~45 min · CORE
⛔ FRIDGE-WALL CAUTION (read first). The cabinet wall has the fridge on the other side. Before driving any screw, measure the wall depth and your screw length so nothing pokes through toward the fridge. If in doubt, use shorter screws or a surface-mount bracket. Grown-up does all drilling.
Hold the VESA bracket in portrait orientation against the spot. Check it's level and that the screen will sit at a good viewing height.
Mark the holes. Measure wall depth vs screw length (fridge caution!). Use anchors rated for the screen's weight. Drill + fix the bracket (grown-up).

Cyborg is on the wall, powered through one switch, cables tidy, fan breathing, and you can still reach the SD/USB. The calendar glows in portrait. 🧲
🛟 If it didn't… it won't sit flat / cables too deep Swap straight HDMI/power cables for the right-angle ones from the BOM. Add a slim cable sleeve. Make sure the Pi isn't pressing on any cable bend.
the finished rear: Pi on standoffs, fan air-gap, tidy cable run, and the SD/USB access point.

Mission 9 — Power button & bedtime schedule

Role lead: 🔌 Captain of Cables · ★★☆ · ~25 min · BONUS

Two nice-to-haves: a real off button (so you never yank the plug), and an automatic screen bedtime so the display rests overnight (the setup script already installed the schedule — 10:00 pm off, 6:30 am on).

Power button (GPIO17)

POWER OFF FIRST.
Wire a momentary button across Pin 11 (GPIO17) and Pin 14 (GND). On a 4-leg button, use two diagonal legs. No resistor needed.
Button leg A ─► Pin 11 (GPIO17)
Button leg B ─► Pin 14 (GND)
Press = clean shutdown (a tiny service watches GPIO17).
The shutdown watcher was installed by setup.sh. Make sure it's running: 
systemctl status cyborg-shutdown-button.service --no-pager
Hold the button ~2 seconds. The Pi powers down cleanly (green light blinks the shutdown pattern, then off).
a long press cleanly shuts Cyborg down — no plug-yanking — and the screen turns itself off at night and back on in the morning. 😴
🛟 If it didn't… button does nothing 4-leg buttons connect the wrong legs easily — use diagonal legs (or test with a multimeter). Confirm the wires are on Pin 11 and Pin 14, and that cyborg-shutdown-button.service shows active (running).

Adjust the bedtime hours (optional)

Edit the OnCalendar= times in /etc/systemd/system/cyborg-screen-off.timer and …-on.timer, then sudo systemctl daemon-reload.

🎖️ Badge earned: 🔘 The Big Red Button

Zone 1 — Weather + meal of the day V2 BLUEPRINT

  2. Add your latitude/longitude + an optional meal_of_day source to config.local.toml.
  3. Expose it: add a weather + meal block to the agenda/hub JSON.
  4. Frontend: a top CSS-grid row showing temp, hi/lo, an icon (system font/emoji — no heavy images), and "Tonight: spaghetti 🍝".

🛟 Troubleshooting HQ

Start here whenever something's off. Cyborg's /health page (http://cyborg.local:8765/health) is your best friend — it usually tells you exactly which part is unhappy.

The screen is totally blank / black Open http://cyborg.local:8765/health on your phone. It loads → the brain is fine; it's a display problem: check the HDMI cable, the monitor's selected input, and that the Pi shows a rainbow splash on boot. It doesn't load → the Pi or service is down: check power (undervoltage lightning-bolt), and on a keyboard run systemctl status cyborg-fetch.
It says "waking up / syncing" forever No calendar data yet. Check /health → fetch.last_error. Usually a bad ICS link (must start with https://) or no internet. Paste your link into a browser — it should download a .ics file.
"Running on backup memory" chip won't go away Cyborg can't reach the internet, so it's showing the last saved copy (good — not blank!). Check Wi-Fi: ping -c3 raspberrypi.com. Re-enter Wi-Fi via sudo raspi-config if needed. Once back online it refreshes within a few minutes.
A lightning bolt appears / random freezes Under-voltage. Use the official 5 V/2.5 A supply and a good cable. vcgencmd get_throttled should be 0x0. Avoid powering the speaker or extras from the Pi's USB.
Events are in the wrong time / wrong day Set home_tz correctly in config.local.toml (e.g. America/Chicago), and make sure the DS3231 clock is set (sudo hwclock -w while online). Restart with sudo systemctl restart cyborg-fetch.
The kiosk shows a desktop / mouse cursor / address bar Re-run sudo ./scripts/setup.sh (safe to repeat). It installs the kiosk autostart, hides the cursor (unclutter), and launches Chromium in full-screen kiosk mode.
BOOYAH never plays Open http://cyborg.local:8765/assets/booyah.mp3 — it must play. If it 404s, the file isn't at /opt/cyborg-core/web/assets/booyah.mp3. Then check the speaker power + jack and do_audio 1.
How do I update Cyborg later? Cyborg doesn't auto-update (safer for a family appliance). To update: cd ~/onward-cyborg && git pull, then sudo ./scripts/setup.sh and sudo systemctl restart cyborg-fetch.
